# 👜 Purse Retrieval System

A two-stage computer vision project:

1. **Indexing** (`build_index.py`, run manually): for every photo in `dataset/women/`,
   detect the purse with a pretrained YOLO model, crop it in-memory, embed the crop with
   DINOv2, and store `{embedding, original_image_path}` in a FAISS index. The original
   photo is never modified, cropped-and-saved, or exposed as output — the crop only ever
   exists in memory long enough to compute an embedding.
2. **Retrieval** (`main.py`, Streamlit app): upload a standalone purse photo, it gets
   embedded the same way, and FAISS returns the original full photo(s) of the model(s)
   carrying the closest-matching purse.

This project uses **off-the-shelf, pretrained models only** — no purse dataset, no
fine-tuning, no training. Matching is **closest visual similarity**, not guaranteed
exact physical-purse re-identification.

---

## Project structure

```
purse_retrieval/
├── main.py                 # streamlit run main.py -- launches the search UI only
├── config.py                # every path/threshold/model name lives here
├── build_index.py           # standalone script: detect + embed + index. Run manually.
├── requirements.txt
├── README.md
├── .gitignore
├── core/
│   ├── __init__.py
│   ├── detection.py         # YOLO purse detection + crop + debug helper
│   ├── embedding.py         # DINOv2 model loading + get_embedding()
│   ├── indexer.py           # FAISS save/load/reset
│   └── search.py            # search_purse() -- threshold + margin filtering
├── dataset/
│   ├── women/                # full model photos (indexed)
│   └── purse_queries/        # standalone purse photos (used to search)
└── embeddings/               # generated by build_index.py -- not committed by default
    ├── faiss.index
    └── metadata.pkl
```

`main.py` never touches YOLO or runs detection — it just loads whatever index
`build_index.py` last produced. This keeps the Streamlit app fast to launch and keeps
the (comparatively slow, one-time) indexing step out of your everyday workflow.

## Setup

```bash
# 1. Create and activate a virtual environment
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # macOS/Linux

# 2. Install dependencies (CPU-only by default -- see "Using your GPU" below)
pip install -r requirements.txt
```

This installs the CPU build of PyTorch by default. For a dataset this size (a few
dozen images), CPU inference on YOLOv8n + DINOv2-small takes seconds — there's no real
need for a GPU here unless you scale up to a much larger dataset.

## Usage

### 1. Build the index (run once, or whenever you add new photos)

```bash
python build_index.py
```

This scans `dataset/women/`, detects the purse in each photo, embeds it, and saves the
FAISS index to `embeddings/`. It's **incremental** — if an index already exists, only
photos that aren't in it yet get processed. To force a full rebuild from scratch (e.g.
after changing `EMBEDDING_MODEL_NAME` in `config.py`):

```bash
python build_index.py --reset
```

If some images come back with "no purse detected," the script tells you which ones.
You can dig into why using the debug helper in a Python shell:

```python
from core.detection import load_yolo_model, debug_detections
model = load_yolo_model()
debug_detections("dataset/women/model_10.jpg", model)
```

This prints everything YOLO saw in that image (any class, any confidence), so you can
tell whether it's a threshold problem or the purse genuinely wasn't recognized.

### 2. Run the app

```bash
streamlit run main.py
```

Upload a purse photo from `dataset/purse_queries/` (or anywhere else) and click Search.

## Tuning match quality

Two settings in `config.py` control how strict matching is:

- **`SIMILARITY_THRESHOLD`** — a hard floor a result must clear to be considered at all.
- **`MATCH_MARGIN`** — a result must *also* be within this margin of the single best
  match to be shown. This is what stops loosely-similar results from flooding the
  output when the top match is a clear standout (e.g. 0.85 vs. 0.55) — a fixed
  threshold alone can't distinguish that gap from a case where several genuinely-close
  candidates exist.

If results look too strict (nothing ever matches) or too loose (everything matches),
adjust these two values based on what you actually observe — don't guess blind. A quick
way to see raw numbers before filtering:

```python
from PIL import Image
import config
from core.embedding import load_embedding_model, get_embedding
from core.indexer import load_database

embed_model, embed_processor = load_embedding_model()
faiss_index, metadata = load_database()

query = Image.open("dataset/purse_queries/purse_01.jpg").convert("RGB")
embedding = get_embedding(query, embed_model, embed_processor).reshape(1, -1)
sims, idxs = faiss_index.search(embedding, config.TOP_K)

for sim, idx in zip(sims[0], idxs[0]):
    if idx != -1:
        print(f"{sim:.3f}  {metadata[idx]['path']}")
```

## Using your GPU (optional)

You mentioned CUDA 12.9 is installed and working. For this dataset size it genuinely
won't make a noticeable difference, but if you scale up to a much larger dataset and
want to speed up indexing, swap the CPU PyTorch wheel for a CUDA build:

```bash
pip uninstall torch
pip install torch --index-url https://download.pytorch.org/whl/cu121
```

(Use whichever `cuXXX` tag matches a CUDA version your installed driver supports --
check https://pytorch.org/get-started/locally/ for the current mapping.)

Then set in `config.py`:

```python
DEVICE = "cuda"
```

Everything else in the codebase already reads `config.DEVICE`, so no other changes are
needed.

## Notes on scope

- **Detection**: pretrained YOLOv8n on COCO weights, using the existing "handbag" class
  (id 26) — no custom training or purse-specific dataset required.
- **Embeddings**: DINOv2-small, chosen over CLIP because it's specifically strong at
  instance-level visual similarity ("is this the *same* object") rather than broad
  semantic similarity ("is this generally the same *kind* of object") — CLIP tended to
  score unrelated bags of similar color/material too close to genuine matches.
  Since this project doesn't fine-tune anything, matching is inherently a "closest
  visual match" rather than guaranteed exact physical-purse re-identification.
- **`dataset/purse_queries/`** contains a mix of purses harvested directly from your own
  `dataset/women/` photos (guaranteed to have a true match in the index) and externally
  sourced purse photos (which may legitimately return "no confident match" if that
  exact purse isn't in your indexed photos — that's correct rejection, not a bug).
